1. Project overview & architecture

CICADA-5453 is a trading system with a React frontend and optional Python FastAPI backend. The frontend is a single-page app (Vite + React) that can run in demo mode (no MT5) or connect to MT5 and the neural-network API for backtesting, bot building, and live execution.

  • Frontend: src/ — React app, global TradingStore, no Redux.
  • Backend: python/cicada_nn/ — FastAPI app, NN training, MT5 client, backtest/research servers.
  • Persistence: Frontend uses localStorage key cicada-5453-store; backend persists bots, positions, execution log under checkpoints/.
  • API base URL: VITE_NN_API_URL (default origin:8000). Remote server URL can be set via Server Offload and stored in localStorage; all NN/MT5 requests then use that URL.

Theme & design system

  • Background: #000000
  • Primary text / borders: #00ff00 (green)
  • Accent / warnings: #ff6600 (orange)
  • Error: #ff4444 (red)
  • Scrollbar: #00ff0080 thumb on black
  • Font: JetBrains Mono (400, 500, 600, 700)
  • Animations: eta-glow, backtest-scanline, backtest-cell-active, strategy-feed-in, radar-sweep-rotate

4. Features & user actions

Auth (Login page)

Purpose: Enter app via MT5 or demo.

Fields: User identifier (MT5 account), Pass key (password), Server (optional). Checkbox: “Continue without MT5 (demo mode)”.

Button: “INITIATE SECURE ACCESS”.

Logic: If demo mode → navigate to /dashboard after 800 ms. Otherwise postMt5Connect(username, password, server); on success applyMt5LoginSuccess and navigate after 1 s; on failure show res.message or “MT5 connection failed” / “Connection failed”.

UI: Animated terminal text; matrix-style canvas; error block below form (red border); loading state on button (“AUTHENTICATING...”); bottom strip “AUTHORIZED ACCESS ONLY”.

Validation: Username and password required={!demoMode}; server optional; fields disabled when demo mode.

Dashboard

Purpose: Central hub for instruments, strategies, backtest, bots, portfolio, brokers.

Layout: Sticky header (logo, LIVE/DISCONNECTED badge, time, Bot execution on/off, broker statuses, Logout); status strip (BOTS, STRAT, WIN, P/L, INST); two columns (left: Instrument selector, chart, spread, Backtest, Bot Builder, Bot Registry, Instrument Manager; right: Server Offload, Live Portfolio, Bot Execution Log, Brokers, Trading Modes, Strategy Library).

Actions: Logout → /; toggle “BOT EXECUTION ON/OFF”; clock updates every second; scheduled rebuilds every 60 s for due bots (rebuild + buildBot).

P/L: Total P/L = open (portfolio.totalPnl) + realized (closed trades with contractId); “—” when no data or not finite.

Instruments

  • InstrumentManager: Toggle active/inactive per instrument (row click); “Apply rebuild interval to all” dropdown; table: Symbol, Type, Broker, Spread, Market (open/closed), Rebuild interval, Actions.
  • InstrumentSelector: Chooses which instrument is selected for Bot Builder and chart.

Strategies

  • StrategyLibrary: Search by name/id/category; enable/disable per strategy; “SELECT ALL” / “DESELECT ALL”; display enabled count and category/regimes.

Backtest

BacktestEngine: Select date range, timeframes, regimes, coverage; “Run backtest” or “Run research”. If backend available and offload configured, tries server backtest first; else client-side. Progress and errors shown.

Buttons: Run backtest, Run research; Robust mode / Grid expanded toggles; [RUN], [CANCEL].

Errors (block run): “No active instruments”, “Select at least one timeframe”, “Invalid date range”, “No strategies enabled”, plus API/run errors.

Bot builder

Actions: Rebuild (mark for rebuild), Build (trigger NN build with backtest results), Deploy / Undeploy; select/deselect timeframes (including “Select all” / “Deselect all”), regimes, styles.

State: Bot status (deployed, building, ready, outdated); build progress %; lastError; cold-start notice (warmup scale < 1).

Logic: Build uses train slice only (validation slice not sent); calls postBuild with best results and instrument types; on success stores feature_vector on bot.

Portfolio

  • LivePortfolio: Balance, equity, open positions, P/L; “Sync positions” button; closed trades (with contractId). Intervals: portfolio tick 5 s, balance sync 10 s, positions sync 15 s when connected and have positions.
  • Sync: syncBrokerPositions() — load from Deriv / eXness API / MT5; only setPositions used for sync; new bot-driven positions go through addPositionWithRiskCheck.

Brokers

BrokersManager: Three default brokers: Deriv, eXness API, MT5 add-on. Per broker: Connect, gear (edit credentials), instrument count.

Deriv: App ID + Token; “Refresh synthetics”, “Add all” from validation.

eXness: API key, optional Base URL.

MT5: login, password, server.

“Add broker”: Opens form (name, type, credentials). Inline forms for credentials (no separate modal); editingId state for which broker is being edited.

Server offload

  • ServerOffload: Username, IP, password, port; “Connect” runs testServerConnection with Basic auth; on success saves URL to localStorage and all NN/MT5 requests use it. “Use this machine” clears remote URL.

6. Edge cases & validation

Form validation

  • Login: Username and password required when not demo mode; server optional. Trim on submit.
  • ServerOffload: Username, IP, password required; port default 8000; URL built from host/port; invalid port clamped to 1–65535.
  • Brokers: Deriv needs App ID + Token; eXness API key; MT5 login + password (server optional). Connect only when credentials present; otherwise open “edit” form.
  • BacktestEngine: Run blocked with explicit messages: no active instruments, no timeframes selected, invalid date range, no strategies enabled.

Backtest bars

Server backtest and research validate bars for all instrument×timeframe keys and minimum length (MIN_BARS_REQUIRED_BACKTEST 10, MIN_BARS_REQUIRED_RESEARCH 200); missing/insufficient bars return 400 and halt (no inference or skip).

Instrument migration

On load, bots pointing to removed instrument IDs are reassigned to first valid instrument; backtest results for removed instruments are dropped.

Deriv synthetics

Only certain groups (e.g. Volatility, Crash/Boom, Jump, Step, Range Break, World) are in the registry; some Volatility/DEX/Drift symbols may not be returned for the account and are not offered.

Out-of-sample & build

Train slice (e.g. first 80% by time) used for build; validation slice not sent to training. Regime at inference: Predict called with regime from detectRegime(bars) and timeframe so decisions are regime-aware.

7. Risk & execution rules

All bot-driven position opens must use addPositionWithRiskCheck (in portfolio.ts). It calls tryOpenPosition (in risk.ts), which:

  • Ensures finite equity, checks drawdown, max positions per instrument, correlated exposure.
  • Runs risk library for the trade scope (scalp/day/swing/position); first failing rule returns allowed: false with ruleId and ruleName.

Risk library (riskLibrary/index.ts): 50+ rules in categories — drawdown, position_limit, exposure, volatility, time, capital, daily_loss, concentration, correlation, sizing, regime, instrument, compliance, liquidity. Filtered by scope; evaluated in order; first failure blocks.

Concentration: “Max single 20%” is skipped for the first position so the first trade is not blocked.

Size: Zero or invalid (non-finite or ≤ 0) after Kelly/warmup is rejected; only finite size > 0 is added.

Drift detection

After ≥10 closed trades per bot, compare live win rate and profit factor to backtest (drift.ts). If live win rate ≥15 pp below backtest or live profit factor < 0.7 (with backtest ≥ 1.2), call setDriftDetected(botId, reason) and schedule early rebuild. Constants: DRIFT_MIN_CLOSED_TRADES 10, DRIFT_WIN_RATE_THRESHOLD_PP 0.15, DRIFT_PROFIT_FACTOR_MIN 0.7.

Risk library rules (summary)

Rules are filtered by trade scope (scalp/day/swing/position); first failing rule blocks the trade. When a rule blocks, ruleId and ruleName are returned for logging.

8. Error states & messages

  • Login: error state; message from API or “MT5 connection failed” / “Connection failed”; red border block.
  • BacktestEngine: setError(...) for no instruments, no strategies, no timeframes, invalid date range, backtest/research failure.
  • BotBuilder: bot.lastError in red box; build failure message from API.
  • BrokersManager: derivValidationError; connect failure from store.
  • ServerOffload: message with type success/error (username, IP, password, connection result).
  • testServerConnection: 401 “Authentication failed”; 403 “Access denied”; 404 “API not found (check port)”; 5xx “Server error”; timeout “Connection timed out”; network “Server unreachable”.
  • ErrorBoundary: Catches render errors; full-page “Something went wrong” + error message; componentDidCatch logs to console.

9. Loading & empty states

  • Login: Button shows “AUTHENTICATING...” with spinners when isLoading.
  • BotBuilder: Build progress bar and EtaDisplay when status is “building”; “Select an instrument” / “No bot for this instrument” when no selection/bot.
  • BacktestEngine: Progress from store during run; “Run backtest” disabled or feedback during run.
  • LivePortfolio: “No data” when dataSource === 'none'; “Stale — reconnect” when disconnected; “Sync positions” shows syncing/synced/error with timeouts.
  • StrategyLibrary: “No strategies match” when filter returns empty.
  • BrokersManager: Connecting state per broker; instrument count 0 for MT5 until connected.
